PRD: B-Roll Director
LLM-as-Director architecture for AI-assisted footage discovery
Author: Erfan Salehi  |  Version: 0.2 (Architecture rewrite)  |  Date: May 2026
1. Overview
1.1 What changed from v0.1
v0.1 used a fragmented pipeline: split the script into ~315 timestamp slots, ask a chat LLM 
(Groq) to generate keywords for each slot in batches of 5-6, then search YouTube. The LLM 
never saw the whole script at once, never knew what shot it suggested 6 seconds earlier, and 
treated every slot as equally important.
v0.2 inverts the model. The LLM acts as a video director: it reads the entire script in one pass, 
produces a structured shot list with intent, shot type, search queries, and priority per timestamp, 
then optionally curates the search results returned from stock footage and YouTube APIs. The 
LLM is provider-abstracted — Gemini Flash, Groq Llama 3.3, DeepSeek, OpenRouter, or local 
Ollama can all play the role.
1.2 Why this is better
•Whole-script context. The LLM sees narrative flow and can avoid visual repetition (no 
'newsroom shot' four times in eight seconds).
•Fewer API calls. One large call replaces ~50 batched calls. Easier on rate limits, faster 
wall-clock time.
•Richer output per slot. Shot intent, shot type, duration needed, priority — not just 
keywords.
•Provider-agnostic. If Groq's free tier changes or Gemini removes free Flash access, 
swap the provider via config. No code changes downstream.
•Optional curation pass. After search APIs return candidates, the LLM can rank them 
based on the shot intent it originally specified — closing the loop.
1.3 Goals
•Produce a structured shot list for a 10-minute video in under 90 seconds (single LLM 
call).
•Keep total time-to-research under 10 minutes including stock and YouTube lookups.
•Run on free-tier services for 2-3 videos/week without hitting paid usage.
•Allow swapping LLM providers via a single config setting.
•Deliver output files an editor can open in any NLE (txt, srt, optional XML/EDL) without 
installing plugins.
1.4 Non-goals (v1)
•Video-frame-level analysis of clips (visual validation by a multimodal model). Reserved 
for v1.5.
•Auto-downloading clips. Reserved for v1.5.•Plugin-based NLE integration (Premiere/Resolve).
•Multi-language support (English script only).
•Cloud deployment or multi-user accounts.
2. Architecture
2.1 Three stages
Stage 1 — Direct: LLM reads full script + voiceover duration. Returns structured shot list as 
JSON.
Stage 2 — Search: For each shot in the list, query stock footage APIs (Pexels, Pixabay) and 
YouTube (yt-dlp) using the search queries the LLM generated. Pure metadata fetching — no 
LLM calls.
Stage 3 — Curate (optional): Send the candidate metadata back to the LLM in a single 
batched call. LLM ranks the top 3 candidates per shot based on the shot intent it originally 
specified. Outputs final ranked shot list.
2.2 Why three stages instead of one
The naive approach is 'just send everything to the LLM' — but that conflates two different 
problems. Stage 1 is a creative task (interpret the script). Stage 3 is a ranking task (pick the best  
from a list). Splitting them lets us:
•Pause between stages so the editor can review and tweak the shot list before burning 
search-API quota.
•Cache stage 1 output by script hash. Re-runs cost nothing if the script hasn't changed.
•Skip stage 3 entirely on simple videos where stage 2's first result is usually good.
•Use different models per stage if useful (e.g. cheap Flash-Lite for Stage 3 ranking, better  
model for Stage 1 directing).
3. LLM Provider Abstraction
3.1 The interface
All LLM calls go through a thin internal interface. The application code never imports a vendor 
SDK directly. A single config file picks the active provider:
# config.yaml llm:   provider: gemini   # or: groq, openrouter, deepseek, 
ollama   model: gemini-3-flash-preview   api_key_env: GEMINI_API_KEY   
max_tokens: 16000   temperature: 0.4
Adding a new provider = adding one Python file (~50 lines) implementing two methods: 
complete(prompt) and complete_json(prompt, schema). Everything else is shared.

3.3 Default and fallback chain
If the configured provider returns an error or hits a rate limit, the tool tries the next in the fallback  
chain rather than failing the run:
1.Primary: Gemini 3 Flash (largest free quota, best whole-script handling)
2.Secondary: Groq Llama 3.3 70B (fast, good fallback)
3.Tertiary: OpenRouter free route (last-resort)
Fallback is configurable. A user with no Gemini key can set Groq as primary.

markers on the timeline with descriptions, no plugin install required.
5. The Director Prompt (Stage 1)
5.1 Inputs to the prompt
•Full script text•Voiceover duration in seconds
•Intro duration setting (default 30s) and pacing rules (1 shot/sec for intro, 1 shot/2sec for 
body)
•Optional style hints from the user
5.2 What the LLM is asked to do
4.Estimate timing: words/second based on script length and audio duration.
5.Generate a shot list covering every required slot. Compute timestamps.
6.For each slot, produce: shot intent, shot type, 2-3 visual concept search queries, 
duration needed, priority (high/medium/low).
7.Avoid visual repetition across adjacent slots. The model has the whole script — it should  
use it.
8.Mark slots that should stay on the talking head (no B-roll) with priority='none' and empty 
queries — based on phrases like 'as I mentioned earlier' or direct address moments.
9.Return strict JSON conforming to the schema. No prose, no markdown.
5.3 Prompt design principles
•Visual concepts, not literal keywords. Bad: "economy 2008". Good: "stock market 
crash trading floor panic", "newspaper headlines recession 2008".
•Shot type vocabulary. Constrain the LLM to a fixed set: wide, medium, close-up, 
extreme close-up, aerial, abstract, graphic. Keeps output usable downstream.
•Priority is meaningful. 'high' = key emotional or narrative moments where the right shot  
really matters. 'low' = filler. Used in Stage 3 to decide where to spend curation tokens.
•Few-shot examples in the prompt. Three worked examples (a typical educational 
sentence, an emotional beat, a transition phrase) anchor output quality.
5.4 Token budget
•Input: ~500 token system prompt + ~2,000 token script (10-min video) + ~200 token 
style hints = ~2,700 tokens.
•Output: ~315 slots × ~40 tokens per slot = ~12,000-14,000 tokens.
•Total: ~16,500 tokens per video. Fits in every provider's context. Free on Gemini, ~$0.04  
on paid Gemini Flash.
6. The Curator Prompt (Stage 3, optional)
6.1 What it does
After Stage 2 returns a list of candidate clips per slot (with title, tags, duration from 
Pexels/Pixabay/yt-dlp), the curator pass sends batches of slots back to the LLM to rank 
candidates. The LLM compares each candidate's metadata to the shot intent it generated in 
Stage 1 and picks the best 3.
6.2 What it does NOT doIt does not watch the video. No frame analysis. No video tokens. Pure text-on-text reasoning 
over metadata. This is the difference between Stage 3 (cheap, free-tier-friendly) and v1.5's 
video validation (paid, multimodal).
6.3 Cost
•Input per batch: 10 slots × (shot intent + 10 candidates × 50 tokens metadata) = ~6,000 
tokens.
•Output per batch: 10 slots × ranking = ~500 tokens.
•Total per video: ~32 batches × 6,500 tokens = ~210K tokens.
•Free on Gemini Flash. ~$0.30 on paid Gemini Flash. Skippable entirely if results from 
stage 2 look good.
7. User Interface (Streamlit)
7.1 Single-page flow
10.Setup (collapsible). LLM provider dropdown. API key inputs for active provider, Pexels, 
Pixabay. 'Test connection' button per provider.
11.Upload. Script (.txt) and voiceover (.mp3/.wav/.m4a). Show detected duration and word 
count.
12.Settings. Intro duration (default 30s). Style hints (free text). Shot pacing override.
13.Direct (Stage 1). 'Generate shot list' button. Single LLM call, ~30-90 sec. Progress 
shown. Results displayed as an editable table — every field can be tweaked inline.
14.Search (Stage 2). 'Find footage' button. Parallel calls to Pexels, Pixabay, yt-dlp. 
Progress bar.
15.Curate (Stage 3, optional). 'Rank with AI' button. Greyed out if user wants to skip.
16.Download. Buttons for shot_list.json, shot_list.txt, timing.srt, download_list.txt, 
markers.fcpxml. Plus 'Download all (.zip)'.
7.2 Design principles
•Stages are explicit and pause-able. The editor controls when to spend API quota.
•Every output is editable before download.
•Provider/key changes don't require restarting the app.
•Persistent local cache keyed by script hash. Re-running the same video is free and 
instant.

9. Cost & Quota Analysis
9.1 Per video, with Gemini 3 Flash (free tier)
•Stage 1: 1 call, ~16K tokens. Free.
•Stage 2: ~315 stock API calls + ~315 yt-dlp searches. Free (Pexels, Pixabay, yt-dlp).
•Stage 3 (optional): ~32 calls, ~210K tokens. Free.
•Total: free. Comfortably within free-tier limits for 2-3 videos/week.9.2 Per video, with Groq Llama 3.3 70B (free tier)
•Stage 1: 1 call, ~16K tokens — but Groq's TPM cap is 6,000/min. Single call exceeds 
this.
•Mitigation: Split Stage 1 into 3-4 sub-calls (script chunked into 2-3 minute segments) 
when using Groq. Each call ~5K tokens, fits in TPM budget. Costs an extra ~30 seconds  
total wall time.
•Stage 3: ~32 calls, ~210K tokens spread across the run. Fine on TPM.
•Total: free, ~9-12 minutes wall-clock per video.
9.3 Per video, paid (DeepSeek V4 Flash, ~$0.18/M)
•Stage 1: ~16K tokens × $0.18/M = ~$0.003
•Stage 3: ~210K tokens × $0.18/M = ~$0.04
•Total: ~$0.05 per video. If free tiers ever break, paid fallback is trivially cheap.

12. Success Metrics
•Time-to-shot-list (Stage 1 only): under 90 seconds for a 10-min video on Gemini 
Flash, under 3 minutes on Groq.
•Time-to-full-research: under 10 minutes for a 10-min video including Stage 2.
•Shot-intent accuracy: editor rates >70% of shot intents as 'a real director would have 
written this' on first 5 videos.
•Top-pick usability: editor uses the #1-ranked candidate as-is for >50% of slots, without 
picking #2 or #3.
•Visual repetition: manually count repeated shots within 8-second windows; target <5% 
of slots.
•Provider portability: can swap from Gemini to Groq via config and produce comparable  
output, no code change.
•Adoption: at least 1 editor running it independently within 2 weeks of pilot.13. v1.5 and Beyond
13.1 v1.5 — Visual validation
Add a multimodal pass for high-priority slots. The LLM (Gemini Flash, OpenAI 4o-mini, or 
Claude) watches the top candidate clip and confirms it matches the shot intent, returning a best-
segment timestamp (e.g. 'the relevant shot is at 02:14-02:22'). Combined with yt-dlp's --
download-sections flag, this enables auto-trim downloads — only the relevant 8 seconds, not 
the full clip.
•Cost: ~$0.30-0.50 per video on paid Gemini Flash, applied only to ~30 high-priority slots.
•Saves the editor from scrubbing 50 downloaded clips looking for the right moment.
13.2 v2.0 — Auto-download and pre-trim
•Tool downloads top picks directly into a project folder using Pexels/Pixabay direct URLs 
and yt-dlp --download-sections.
•Folder structure: project_name/00_00_02_intro.mp4, 00_00_03_presenter.mp4, etc.
•Editor drags the folder into Premiere/Resolve and clips are pre-named, pre-trimmed, 
ready to drop on the timeline.
13.3 v2.5+ — Optional NLE integration
•Once the Premiere MCP ecosystem stabilizes, add a one-button 'Send to Premiere' that 
places clips on the timeline at correct timestamps.
•Treat as opt-in — the FCPXML output covers 80% of the workflow benefit without plugin 
install.
13.4 Other deferred ideas
•Multi-language scripts (Persian first, given Tovo Studio context).
•Channel-level visual consistency (pass previous shot lists into the director prompt).
•Alternative shot lists ('show me 3 directors' takes').
•Editor feedback loop — track which top picks the editor actually uses and feed back into 
prompt tuning.
